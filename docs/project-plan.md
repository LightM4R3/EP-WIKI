# EPWIKI 초기 구상안

## 1차 목표

EPWIKI는 명일방주: 엔드필드의 정보를 한곳에서 조회하고, 전투 세팅을 조합해 결과를 비교하며, 위기 협약 제약을 계획할 수 있는 Single View Point 도구를 목표로 한다.

초기 제공 영역은 다음 세 가지다.

1. 오퍼레이터, 무기, 장비, 세트 옵션의 기본 정보 카탈로그
2. 레벨, 정예화, 스킬 레벨, 잠재 능력, 재능, 무기 레벨, 무기 기질, 장비 옵션을 반영하는 피해량 계산기
3. 위기 협약 제약의 점수, 선행 조건, 충돌 조건, 선택 목록 관리

## 추천 데이터 구조

게임 데이터는 UI 컴포넌트에 직접 박지 않고 `src/data` 또는 추후 별도 데이터 패키지에 둔다.

- `Operator`: 기본 스탯, 역할, 피해 타입, 재능, 스킬, 검증 상태
- `OperatorDataset`: 실제 입력용 JSON. 이름, 출시 정보, 클래스, 속성, 무기 타입, 진영, 종족, CV, 레벨별 스탯, 스킬, 재능, 잠재, 공업 스킬, 정예화 재료, 추천 장비, 출처를 함께 저장
- `Weapon`: 호환 역할, 공격력, 무기 기질, 레벨 성장 테이블
- `EquipmentSet`: 세트 조각 수, 고정 옵션, 퍼센트 옵션, 세트 효과
- `DamageBuild`: 계산기에 들어가는 모든 선택 상태
- `CrisisRisk`: 제약 점수, 카테고리, 선행 조건, 충돌 조건

실제 데이터 입력 포맷은 `src/data/game/operator.schema.json`과 `src/data/game/operators.seed.json`으로 먼저 잡았다. 데이터가 늘어나면 JSON Schema 검증 또는 Zod 런타임 검증을 붙이고, 이후 SQLite 기반 정적 데이터 빌드로 옮기는 방향이 좋다.

## 계산기 설계

피해량 계산은 화면과 분리된 순수 함수로 둔다.

- UI: 사용자가 빌드 상태를 선택한다.
- Model: `DamageBuild`를 입력받아 `DamageResult`를 반환한다.
- Tests: 공식이 확정될 때마다 대표 케이스를 고정해 회귀 테스트를 추가한다.

현재 `src/features/damage-calculator/damageModel.ts`의 `scaffold-v0` 공식은 화면 검증용 임시 계산식이다. 실제 전투 공식, 버프 중첩 규칙, 방어/저항/피해 증가 그룹이 확인되면 이 파일부터 교체한다.

## 위기 협약 설계

제약은 단순 체크박스가 아니라 관계 그래프에 가깝다.

- `prerequisites`: 선택 전에 필요한 하위 제약
- `conflictsWith`: 동시에 선택할 수 없는 제약
- `score`: 총점 계산에 쓰이는 위험 점수
- `category`: 필터와 그룹 표시 기준

다음 단계에서는 누락 전제를 자동 선택할지, 경고만 표시할지 정책을 정해야 한다.
